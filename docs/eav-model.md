# EAV-модель «Атрибуты фильмов» (CinemaDB)

Дополнение к [logical-model.md](logical-model.md): гибкое
хранение разнотипных атрибутов фильма (рецензии, премии, важные и служебные
даты, точные числовые показатели, приближённые средние оценки) без
изменения нормализованной схемы кинотеатра.

## 1. Зачем EAV

У фильма может быть произвольный, заранее не известный и растущий набор
характеристик разных типов: текстовые рецензии, логические признаки премий,
даты, числовые показатели. Добавлять для каждой характеристики отдельную
колонку в `movies` означало бы постоянные миграции схемы и NULL-разреженную
таблицу. EAV выносит эти характеристики в отдельную подсхему из 3 таблиц,
не трогая нормализованную схему кинотеатра.

## 2. Сущности

| # | Сущность | Назначение |
|---|----------|------------|
| 1 | `attribute_types` | Справочник типов атрибутов: код, отображаемое имя, тип данных, флаг «для маркетинга» |
| 2 | `attributes` | Конкретные атрибуты (например, «Оскар», «Мировая премьера»), привязанные к типу |
| 3 | `attribute_values` | Значения атрибутов для конкретных фильмов, по 5 типизированным колонкам |
| — | `movies` | Переиспользуется из основной схемы (см. logical-model.md) |

## 4. Типы данных и специфика float

Каждому `attribute_types.data_type` соответствует ровно одна типизированная
колонка в `attribute_values`:

| data_type | Колонка | Пример атрибута |
|---|---|---|
| text | value_text | Рецензия критика |
| boolean | value_boolean | Оскар (получен/нет) |
| date | value_date | Мировая премьера / дата начала продажи билетов |
| number | value_number | Рейтинг критиков (IMDb), кассовые сборы |
| float | value_float | Средняя пользовательская оценка |

Числовых типов два, сознательно, под разную специфику значений —
недостаточно один раз решить «float или decimal» и на этом остановиться:
нужно понимать, для каких значений годится каждый.

**`value_number` — `DECIMAL(12,3)`, точная десятичная арифметика.**
Используется там, где округление недопустимо: официально публикуемый
рейтинг, кассовые сборы. `FLOAT`/`DOUBLE` — двоичные типы с плавающей
точкой (IEEE 754); многие десятичные дроби (например, `8.8`, `7.1`) не
имеют точного представления в двоичном виде, из-за чего накапливается
ошибка округления, а сравнения на равенство (`value = 8.8`) могут
неожиданно не срабатывать после серии арифметических операций. `DECIMAL`
хранит число как точное десятичное значение — без этих сюрпризов.

**`value_float` — `FLOAT`, двоичная плавающая точка, сознательно.**
Не всякое число обязано быть `DECIMAL` — там, где значение уже само по себе
приближённое (например, «Средняя пользовательская оценка», вычисленная как
среднее по множеству голосов), точность IEEE 754 ожидаема и достаточна, а
компактность/скорость `FLOAT` — уместный выбор. Специфика хранения и
использования именно этого типа, которую нужно учитывать в запросах поверх
`value_float`:
- **не сравнивать на равенство напрямую** (`value_float = 7.1` может не
  сработать, даже если значение было записано буквально как `7.1`) —
  сравнивать через допуск: `ABS(value_float - 7.1) < 1e-4`;
- **не выводить сырое значение наружу** — округлять при отображении
  (`ROUND(value_float, 1)`), иначе пользователь может увидеть что-то вроде
  `7.099999904632568` вместо `7.1`;
- **не суммировать много таких значений без осознания** накапливаемой
  погрешности (`SUM()`/`AVG()` по тысячам строк `FLOAT` дрейфует иначе, чем
  по `DECIMAL`).

Обе иллюстрации (наивное сравнение vs сравнение с допуском, округление для
отображения) — рабочий пример в `sql/06_eav_queries.sql`, а не только текст
здесь: реальная демонстрация специфики float на реальных данных, а не
теоретическое рассуждение о том, почему float избегается.

У любой корректной строки `attribute_values` заполнена ровно одна из 5
типизированных колонок — та, что соответствует `data_type` атрибута.
Это не проверяется триггером или CHECK-констрейнтом (см. раздел 6) —
целостность проверяется отдельным SQL-запросом.

**Чтение `DECIMAL` из PHP (на будущее, когда появится приложение):**
`DECIMAL` решает проблему точности только пока значение остаётся в SQL или
в PHP-строке — сама конвертация не бесплатна и теряет точность в обе
стороны:

- PDO по умолчанию возвращает `DECIMAL`-колонки PHP-**строкой**, а не
  числом — это сделано намеренно, чтобы не терять точность на границе
  SQL↔PHP.
- Если эту строку привести к `float` (`(float) $value`), точность
  теряется обратно — `float` в PHP это тот же двоичный IEEE 754, что и
  `FLOAT`/`DOUBLE` в MySQL, с теми же приближениями (`0.1` не представимо
  точно). Для точной арифметики над значением после чтения нужно работать
  либо со строкой напрямую, либо через BCMath/GMP, а не приводить к
  `float`.
- Обратное направление тоже не бесплатно: PHP `int` (64-битный, до
  `9223372036854775807`) не помещается в целую часть `DECIMAL(12,3)`
  (максимум 9 цифр, `999999999`) — для целочисленных величин, которым
  нужна такая размерность, подходящая колонка — `BIGINT`, а не `DECIMAL`;
  `BIGINT SIGNED` как раз совпадает по диапазону с PHP `int` без потерь.

В отличие от `DECIMAL`, `value_float` конвертируется в PHP `float` и обратно
**без дополнительной потери точности на самой границе** — оба представления
одно и то же IEEE 754 двоичное число. Но это не значит, что проблема исчезла:
неточность, уже заложенная в само хранимое значение (раздел выше), просто
переносится в PHP как есть — те же правила (сравнение через допуск,
округление для отображения) продолжают действовать и на стороне PHP.

## 5. Views

### `v_marketing_attributes`
Колонки: `movie_id`, `movie_title`, `attribute_type`, `attribute_name`,
`value_display`. Разворачивает все атрибуты с `attribute_types.is_marketing =
TRUE` (рецензии, премии, важные даты, числовые показатели — служебные даты
исключены, они не для печати/показа зрителю), значение всегда приведено к
тексту через `CASE` по `data_type`.

### `v_service_tasks`
Колонки: `movie_id`, `movie_title`, `tasks_today`, `tasks_in_20_days`.
Берёт только атрибуты типа `service_date`, агрегирует названия атрибутов
(`GROUP_CONCAT`) отдельно для тех, чья `value_date` равна `CURDATE()`, и
отдельно — `CURDATE() + INTERVAL 20 DAY`. View пересчитывается динамически
при каждом обращении.

## 6. Проверка целостности

EAV-схема не ограничивает на уровне СУБД, какая типизированная колонка
`attribute_values` должна быть заполнена для атрибута данного типа — это
осознанное решение (см. `docs/superpowers/specs/2026-07-20-eav-movie-attributes-design.md`,
решение №4): вместо триггеров используется отдельный проверочный SQL-запрос
в `sql/06_eav_queries.sql`, который находит строки с:

1. неверным количеством заполненных типизированных колонок (0 или ≥2, из 5);
2. заполненной колонкой, не соответствующей `data_type` атрибута.

На корректных данных запрос возвращает 0 строк.

## 7. ER-диаграмма

```mermaid
erDiagram
    movies ||--o{ attribute_values : "имеет значения"
    attributes ||--o{ attribute_values : "определяет"
    attribute_types ||--o{ attributes : "группирует"

    attribute_types {
        int attribute_type_id PK
        varchar code
        varchar name
        enum data_type
        boolean is_marketing
    }
    attributes {
        int attribute_id PK
        int attribute_type_id FK
        varchar name
    }
    attribute_values {
        bigint value_id PK
        int movie_id FK
        int attribute_id FK
        text value_text
        boolean value_boolean
        date value_date
        decimal value_number
        float value_float
    }
    movies {
        int movie_id PK
        varchar title
    }
```
